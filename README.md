# Notification Service (Transactional Outbox)

REST API сервис для гарантированной доставки уведомлений (Email/Telegram) с поддержкой асинхронной генерации отчетов.

## Требования к окружению

- **Docker** 24.0+
- **Docker Compose** v2.20+
- **PHP** 8.3 (внутри контейнера)

## Быстрый запуск

1. **Настройка окружения**:
   ```bash
   cp .env.example .env
   ```

2. **Запуск и сборка**:
   ```bash
   docker compose up -d --build
   ```
   *Сборка автоматически выполнит установку зависимостей, генерацию ключа, миграции и наполнение базы (seed) через специализированный Init-контейнер.*

3. **API доступно по адресу**: `http://localhost:8080/api`

## Мониторинг фоновых процессов

- **Outbox Relay** (мониторинг таблиц):
  ```bash
  docker compose logs -f outbox-relay
  ```

- **Queue Worker** (выполнение задач и генерация отчетов):
  ```bash
  docker compose logs -f worker
  ```

## Архитектурные решения

### 1. Transactional Outbox Pattern
Решение принято для обеспечения надежности доставки. Уведомление сохраняется в БД в одной транзакции с бизнес-логикой. Это гарантирует, что данные не потеряются при сбоях Redis или брокера сообщений. Отдельный процесс Relay сканирует таблицы и перекладывает задачи в очередь.

### 2. Идемпотентность (X-Idempotency-Key)
Для защиты от дублирования при сетевых сбоях внедрено Middleware. Оно проверяет уникальный ключ в заголовках запроса и кэширует ответ в Redis на 24 часа. Повторные запросы с тем же ключом получают результат из кэша без запуска процесса обработки.

### 3. Гарантия доставки и Retry Policy
Внешние API (Telegram/SMTP) могут быть временно недоступны. Реализована политика повторов с экспоненциальной задержкой. Если после 5 попыток статус не сменился на "sent", запись помечается как "error" для последующего анализа.

### 4. Конкурентная обработка (SKIP LOCKED)
Команда Relay спроектирована для работы в многопоточном режиме. Использование SQL-конструкции FOR UPDATE SKIP LOCKED на уровне PostgreSQL позволяет запускать несколько экземпляров Relay параллельно, исключая захват одной и той же записи разными процессами.

### 5. Безопасность идентификаторов (UUID)
В системе используется разделение ID. Системные инкрементальные ключи (BigInt) применяются только для внутренних связей в БД, тогда как во внешних API используются исключительно UUID (v4). Это скрывает реальный объем данных в базе и защищает от перебора ID.

## Что можно улучшить в продакшн-версии

1. Разделение базы данных: Настройка Read/Write реплик для снятия нагрузки на чтение с основной БД.
2. Laravel Horizon: Подключение панели мониторинга и системы приоритезации очередей (например, мгновенные уведомления в приоритете над тяжелыми отчетами).
3. Мертвые очереди (DLQ): Создание механизма для уведомлений, исчерпавших лимит попыток доставки, для ручного разбора.
4. Оптимизация пагинации: Реализовал поддержку динамического параметра `per_page`, что критично для мобильных приложений, но в будущем можно перейти на Cursor Pagination для работы с гигантскими объемами данных.
5. Расширенный мониторинг: Интеграция Prometheus/Grafana для отслеживания лага Outbox Relay в реальном времени.
6. Оптимизация проверки существования сущностей: Чтобы не нагружать БД частыми запросами exists или find (например, при проверке пользователя перед генерацией отчета), можно использовать фильтры Блума или хэш-таблицы в Redis. Это позволит мгновенно отсекать запросы с несуществующими ID/UUID.
7. Паттерн «Стратегия» для отчетов: Сейчас генерация жестко привязана к CSV. В будущем можно внедрить интерфейс ReportGeneratorInterface, что позволит прозрачно добавлять новые форматы (PDF, Excel, JSON) без изменения логики контроллера.
